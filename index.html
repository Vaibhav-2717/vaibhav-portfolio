04 · HOW I WORK

From requirement to production, not just code.

The same four-step approach I use whether I'm scoping a new CRM module or debugging a production incident.

01

Understand the domain

Requirement analysis and architecture review before writing a line of code — talking to product and QA to find the actual constraint, not just the stated one.

05 · PROJECTS

Selected engineering work.

Each project below is a real system I've built or owned — the problem it solved, how it's architected, and why.

Enterprise CRM Platform

Flagship · Current
Problem

Enterprises needed a configurable, multi-user CRM covering leads, contacts, organizations, activities, and communication — with automation and multi-channel notifications, not a rigid off-the-shelf tool.

Architecture

NestJS + TypeScript service layer; MySQL for transactional core, Neo4j for relationship data, MongoDB for flexible documents, Redis for cache/sessions, Kafka/RabbitMQ/BullMQ for async workflows.

Key Decision

Introduced Neo4j specifically for CRM relationship data (contacts ↔ organizations ↔ activities) — relational joins across deeply connected entities don't scale as cleanly as native graph traversal.

Outcome

Platform serving 50,000+ users across 10+ modules; reduced API latency and increased throughput by moving notification and messaging work onto queues.

MINT — Wealth Management Platform

FinTech
Problem

A wealth management ERP needed secure import of financial data from multiple external sources, plus mutual fund transaction logic — mergers, demergers, folio processing.

Architecture

Node.js + Express.js backend on MySQL, with third-party authentication integrations and SMS-based OTP / two-factor login flows.

Key Decision

Built a dynamic Access Control List (ACL) framework instead of hardcoded roles, so permissions could be configured per business requirement — recognized with the Solutionist Award.

Outcome

Enabled investment and portfolio reporting for financial advisors and customers on a configurable, auditable permission model.

Payment Gateway Platform

Payments
Problem

Merchants needed customizable convenience and processing fee logic across more than one payment gateway provider.

Architecture

Node.js, Express.js, and MySQL backend integrating Razorpay and PayU behind a single configurable API.

Key Decision

Made fee logic configurable per merchant rather than hardcoded, so onboarding a new merchant doesn't require a code change.

Outcome

Secure payment processing with callback handling supporting multiple merchants and fee structures.
